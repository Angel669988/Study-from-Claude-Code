## 一、先统一目标函数（不统一就别做实验）

### 主指标（North Star）
- 一次完成率：用户无需二次追问即可拿到可用结果。

### 护栏指标（Guardrails）
- 成本：单任务 token 成本（input/output/cache）
- 风险：误放行率、误拦截率
- 效率：平均回合数、工具调用数
- 体验：平均时延（可选）

### 通过线（建议）
- 主指标提升 >= 5%
- 护栏不劣化（任一护栏恶化则不通过）

---

## 二、Claude Code 源码如何映射到 Harness

把“业务现象”直接映射到“可调杠杆”：

- 为什么这次没完成？
  - 看 runtime 迭代闭环：`rust/crates/runtime/src/conversation.rs`
- 为什么该拦没拦 / 该放没放？
  - 看权限与钩子：`rust/crates/runtime/src/permissions.rs` + `rust/crates/runtime/src/hooks.rs`
- 为什么成本失控？
  - 看使用量与压缩：`rust/crates/runtime/src/usage.rs` + `rust/crates/runtime/src/compact.rs`
- 为什么策略不稳定？
  - 看上下文组装：`rust/crates/runtime/src/prompt.rs`
- 为什么版本迭代没节奏？
  - 看外环能力：`rust/crates/compat-harness/src/lib.rs` + `src/parity_audit.py`

一句话：
Harness 不是某个模块，而是“目标函数 + 运行时杠杆 + 外环治理”的组合工程。

## 三、第一周实验（你现在就能做）

## 实验主题
提高“一次完成率”，先只动 1 个杠杆：`Query Rewrite（中文）`

## Day 1：建基线
- 准备 30 条真实中文 query（尽量覆盖 3 类任务）
- 跑当前版本，记录：一次完成率 / 回合数 / 成本 / 误拦截

## Day 2-4：只改一件事
- 改写规则：
  - 保留原句
  - 追加能力词（任务目标、输出格式、约束）
  - 0 命中触发二次改写

## Day 5：同样本回放
- 同一批 30 条 query 重跑
- 比较“改前 vs 改后”

## Day 6：失败样本归因
- 每条失败归类到：Prompt / Route / Permission / Runtime / Cost

## Day 7：评审结论
- 只回答 1 个问题：
  - “这次改动是否提升了一次完成率，且护栏不劣化？”
